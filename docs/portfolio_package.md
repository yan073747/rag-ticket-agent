# 作品集包装材料

## 展示目标

这个项目要让 HR 和面试官在 3 分钟内看明白三件事：

1. 这不是普通聊天机器人，而是完整的企业知识库 RAG + 工单 Agent。
2. 项目包含文档上传、文本切分、Embedding、向量检索、来源引用、低置信度转人工、后台记录和评估指标。
3. 你能解释清楚系统为什么这样设计，以及如何验证它没有乱答。

## 演示视频脚本

建议视频控制在 2 到 3 分钟。

### 开场 15 秒

大家好，这是我的作品集项目：企业知识库 RAG + 工单客服 Agent。

它不是普通聊天机器人，而是模拟企业智能客服落地场景：先上传企业制度文档，再做文本切分、Embedding、向量检索，最后基于 RAG 回答问题。如果系统判断知识库无法可靠回答，会自动创建工单交给人工客服处理。

### 第一步：重置演示数据 15 秒

这里先点击“重置演示数据”，把文档、问答、工单和向量索引清空。

这样做是为了保证每次演示都从干净状态开始，避免旧数据影响检索结果。

### 第二步：上传文档入库 30 秒

我上传三份企业制度文档：考勤管理制度、财务报销制度、信息安全制度。

系统会自动完成三个动作：保存文档、切分文本块、写入 Chroma 向量库。这里的切分长度和重叠长度可以调整，重叠是为了避免关键信息刚好被切断。

### 第三步：文档内问题 40 秒

现在问一个文档里有答案的问题：员工入职满 10 年但不满 20 年，每年有多少天年假？

系统返回了答案、置信度和来源片段。这里重点是来源引用，面试官可以看到答案不是模型凭空生成的，而是来自知识库中的具体文本块。

### 第四步：文档外问题 40 秒

再问一个文档里没有的问题：公司 2024 年年度营收是多少，目前有多少名员工？

这类问题不在三份制度文档里。系统不会强行回答，而是显示低置信度并自动创建工单。下面的候选片段只给人工客服参考，不作为可直接采信的答案来源。

### 第五步：后台和指标 30 秒

最后看后台记录。这里能看到上传文档、历史问答和工单列表。

运营指标里可以看到文档数、问答记录数、工单数、平均置信度和转人工比例。这体现了 RAG 系统的效果评估闭环。

### 结尾 15 秒

这个项目覆盖了企业 RAG 应用的核心链路：知识库构建、检索增强生成、置信度控制、人工兜底和运营评估。

后续可以把本地 Embedding 替换成 OpenAI Embedding 或 BGE，把 SQLite 升级为 PostgreSQL，并接入真实工单系统。

## 简历项目描述

### 一句话版本

企业知识库 RAG + 工单客服 Agent：基于 FastAPI、Chroma、SQLite 和前端工作台，实现文档上传、文本切分、Embedding、向量检索、来源引用、低置信度转人工和后台评估指标。

### 简历项目经历版本

企业知识库 RAG + 工单客服 Agent

- 使用 FastAPI 构建后端 API，完成文档上传、文本切分、Embedding 写入、向量检索、RAG 问答和后台管理接口。
- 使用 Chroma 存储文本块向量，支持基于问题的 Top K 检索，并返回文档 ID、文本块序号和距离等来源信息。
- 设计低置信度转工单流程，当知识库无法可靠回答时自动创建人工客服工单，避免模型编造答案。
- 使用 SQLite 保存文档、文本块、问答记录和工单记录，并实现文档数、问答数、工单数、平均置信度、转人工比例等评估指标。
- 构建前端工作台，支持上传入库、RAG 提问、来源引用展示、候选片段提示、后台记录查看和演示数据重置。
- 编写自动化测试覆盖健康检查、文档上传、文本切分、Embedding、向量检索、RAG 问答、工单和后台指标等核心流程。

### 面试讲解重点

- 为什么 RAG 必须返回来源引用。
- 为什么低置信度要转人工，而不是强行回答。
- 文本切分中的 `chunk_size` 和 `chunk_overlap` 分别解决什么问题。
- Chroma 在项目里负责什么，SQLite 在项目里负责什么。
- 当前本地 Embedding 的优缺点，以及如何替换为真实语义 Embedding。

## 后续优化路线

1. 接入真实 Embedding 模型，提高中文语义召回能力。
2. 使用 LLM 生成更自然的客服话术，而不是直接拼接来源文本。
3. 增加工单状态流转，例如 `open`、`processing`、`resolved`。
4. 增加管理员标注功能，用人工反馈优化评估指标。
5. 使用 Docker Compose 一键启动后端、数据库和前端。
