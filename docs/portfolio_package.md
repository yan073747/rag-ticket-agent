# 作品集包装材料

## 展示目标

这个项目要让 HR 和面试官在 3 分钟内看明白三件事：

1. 这不是普通聊天机器人，而是完整的企业知识库 RAG + 工单 Agent。
2. 项目包含多格式文档上传、文本切分、Embedding、向量检索、DeepSeek 生成、来源引用、低置信度转人工、后台记录和评估指标。
3. 你能解释清楚系统为什么这样设计，以及如何通过批量评测验证它没有乱答。

## 演示视频脚本

建议视频控制在 2 到 3 分钟。

### 开场 15 秒

大家好，这是我的作品集项目：企业知识库 RAG + 工单客服 Agent。

它不是普通聊天机器人，而是模拟企业智能客服落地场景：先上传企业制度文档，支持 TXT、PDF、Word 等格式，再做文本切分、Embedding、向量检索，最后用 DeepSeek 基于检索来源生成客服回复。如果系统判断知识库无法可靠回答，会自动创建工单交给人工客服处理。

### 第一步：重置演示数据 15 秒

这里先点击“重置演示数据”，把文档、问答、工单和向量索引清空。

这样做是为了保证每次演示都从干净状态开始，避免旧数据影响检索结果。

### 第二步：上传文档入库 30 秒

我上传三份企业制度文档：考勤管理制度、财务报销制度、信息安全制度。其中既可以是 TXT，也可以是 PDF 或 DOCX。

系统会自动完成四个动作：保存文档、提取正文、切分文本块、写入 Chroma 向量库。这里的切分长度和重叠长度可以调整，重叠是为了避免关键信息刚好被切断。

### 第三步：文档内问题 40 秒

现在问一个文档里有答案的问题：员工入职满 10 年但不满 20 年，每年有多少天年假？

系统返回了 DeepSeek 生成的客服式回答、置信度和来源片段。这里重点是来源引用，面试官可以看到答案不是模型凭空生成的，而是来自知识库中的具体文本块。

### 第四步：文档外问题 40 秒

再问一个文档里没有的问题：公司 2024 年年度营收是多少，目前有多少名员工？

这类问题不在三份制度文档里。系统不会强行回答，而是显示低置信度并自动创建工单。下面的候选片段只给人工客服参考，不作为可直接采信的答案来源。

### 第五步：后台、指标和批量评测 40 秒

最后看后台记录。这里能看到上传文档、历史问答和工单列表。

运营指标里可以看到文档数、问答记录数、工单数、平均置信度和转人工比例。项目还提供批量评测脚本，可以基于固定问题集生成评测报告，统计通过率、关键词命中率和转人工准确率。这体现了 RAG 系统的效果评估闭环。

### 结尾 15 秒

这个项目覆盖了企业 RAG 应用的核心链路：多格式知识库构建、检索增强生成、置信度控制、人工兜底、后台运营和批量评测。

后续可以把本地 Embedding 替换成 BGE 或 Qwen Embedding，把 SQLite 升级为 PostgreSQL，并接入真实工单系统和权限登录。

## 简历项目描述

### 一句话版本

企业知识库 RAG + 工单客服 Agent：基于 FastAPI、Chroma、SQLite、DeepSeek 和前端工作台，实现 PDF/Word 文档入库、文本切分、向量检索、RAG 问答、来源引用、低置信度转人工、后台记录和批量评测报告。

### 简历项目经历版本

企业知识库 RAG + 工单客服 Agent

- 使用 FastAPI 构建后端 API，完成文档上传、文本切分、Embedding 写入、向量检索、RAG 问答和后台管理接口。
- 支持 TXT、Markdown、PDF、DOCX 多格式企业文档入库，使用 pypdf 和 python-docx 提取正文后进入统一切分流程。
- 使用 Chroma 存储文本块向量，支持基于问题的 Top K 检索，并返回文档 ID、文本块序号和距离等来源信息。
- 接入 DeepSeek Chat Completions API，在高置信度检索结果上生成客服式回答，并对 Markdown 输出做后端清洗。
- 设计低置信度转工单流程，当知识库无法可靠回答时自动创建人工客服工单，避免模型编造答案。
- 使用 SQLite 保存文档、文本块、问答记录和工单记录，并实现文档数、问答数、工单数、平均置信度、转人工比例等评估指标。
- 构建前端工作台，支持上传入库、RAG 提问、来源引用展示、候选片段提示、后台记录查看和演示数据重置。
- 编写批量评测脚本，基于 JSON 问题集生成 Markdown 报告，统计通过率、关键词命中率、转人工准确率和平均置信度。
- 编写自动化测试覆盖健康检查、文档上传、PDF/Word 解析、文本切分、Embedding、向量检索、RAG 问答、工单、后台指标和批量评测等核心流程。

### 面试讲解重点

- 为什么 RAG 必须返回来源引用。
- 为什么低置信度要转人工，而不是强行回答。
- 文本切分中的 `chunk_size` 和 `chunk_overlap` 分别解决什么问题。
- Chroma 在项目里负责什么，SQLite 在项目里负责什么。
- DeepSeek 在项目里负责生成回答，RAG 检索负责提供可信来源，二者不是互相替代。
- 批量评测报告如何证明系统既能回答文档内问题，也能拒答文档外问题。
- 当前本地 Embedding 的优缺点，以及如何替换为真实语义 Embedding。

## 后续优化路线

1. 接入真实 Embedding 模型，提高中文语义召回能力。
2. 增加工单状态流转，例如 `open`、`processing`、`resolved`。
3. 增加管理员登录和权限控制。
4. 增加人工标注反馈，用于评估和优化 RAG 回答质量。
5. 将 SQLite 升级为 PostgreSQL，并部署到云服务器或作品集网站子路径。
